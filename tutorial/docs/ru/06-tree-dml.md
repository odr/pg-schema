# Tree DML: insertJSON и upsertJSON

Цель главы: разобрать запись дерева данных за один запрос и понять алгоритм `upsertJSON`.

## Что важно сразу

- `insertJSON` и `upsertJSON` работают с root + nested children.
- Оба пути используют один SQL-пайплайн записи дерева.
- Ключевое отличие между ними — ограничения на вход.

## `insertJSON`: строгая вставка

Правило:
- на каждом узле дерева должны быть переданы все обязательные поля.

Используйте, когда создаете новые данные и не ожидаете конфликтов по ключам.

## `upsertJSON`: вставка или обновление

Правило для каждой строки:
- либо есть полный mandatory-набор,
- либо есть полный PK (для однозначной идентификации строки).

Это делает `upsertJSON` удобным для синхронизации и повторных запусков.

## Алгоритм upsertJSON (практическая модель)

1. Проверить форму входного дерева на уровне типов.
2. Для каждого узла определить стратегию идентификации (mandatory или PK).
3. Построить SQL для root и всех дочерних веток.
4. Выполнить операцию как один round-trip к БД.
5. Вернуть результат в типизированной форме.

## Returning и ограничение поддерева

Практическое правило:
- возвращаемое дерево должно быть поддеревом входного дерева.

Это снижает риск “невалидного ожидания” результата и проверяется compile-time.

## Почему это важно (perf + transactional)

- Один запрос вместо нескольких ручных шагов.
- Меньше накладных расходов на сетевые round-trip.
- Проще сохранить целостность операции в рамках одной БД-команды.
