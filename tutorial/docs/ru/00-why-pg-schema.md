# Зачем нужен pg-schema

`pg-schema` — это подход к работе с PostgreSQL, где схема БД становится частью типовой модели Haskell-кода.

Главная идея:
- схема базы один раз извлекается в Haskell-модуль;
- дальше `SELECT/INSERT/UPDATE/DELETE` строятся с проверками на этапе компиляции;
- ошибки в именах таблиц, полей и связей ловятся до запуска приложения.

## Что это дает на практике

Обычный риск при работе с SQL в приложении:
- опечатка в названии поля;
- неправильная связь между таблицами;
- несовместимый тип параметра;
- невалидный shape данных при вложенных вставках.

С `pg-schema` эти классы ошибок переносятся из runtime в compile-time.

## Ключевые возможности

- Генерация type-level схемы из живой PostgreSQL БД.
- Типобезопасный `selectSch` с навигацией по связям (`qPath`).
- Вставка/апдейт дерева данных (`insertJSON`/`upsertJSON`) за один запрос.
- Поддержка enum, json/jsonb, массивов (`PgArr`) и сложных ключей.

## Минимальный пример (концептуально)

1. Генерируем модуль схемы `Sch`.
2. Описываем `Ann` для корневой таблицы.
3. Выполняем типобезопасный `selectSch`:

```haskell
-- условный пример
type MyAnn tab = 'Ann 5 CamelToSnake Sch ("tut" ->> tab)

rows <- selectSch (MyAnn "projects") conn $ qRoot do
  qWhere $ "status" =? (ProjectStatus_active)
  qPath "tasks_project_fk" do
    qWhere $ "priority" <? (50 :: Int32)
```
