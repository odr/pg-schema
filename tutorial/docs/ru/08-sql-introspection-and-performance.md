# SQL introspection и производительность

Цель главы: показать, как смотреть generated SQL и как это связано с one round-trip, производительностью и транзакционностью.

## Почему это важно

`pg-schema` не “прячет” SQL:
- можно получить текст запроса,
- проверить параметры,
- понять, почему операция быстрая или медленная.

## Инструменты introspection

Для чтения SQL используйте:
- `selectText`
- `insertText` / `insertJSONText`
- `updateText`
- `deleteText`

Это базовый путь к отладке и объяснению поведения запроса.

## One round-trip: что это дает

Для tree read/write библиотека строит один SQL-запрос.

Практические эффекты:
- меньше сетевых прыжков между приложением и БД;
- ниже latency при сложных nested-операциях;
- проще reasoning о целостности операции.

## Транзакционный аспект

Одна операция к БД не заменяет бизнес-транзакции целиком, но:
- уменьшает число промежуточных состояний между шагами;
- снижает риск “полуобновленного” дерева при ручном многошаговом коде.

## Разбор кейса с массивами (`PgArr`)

Отдельный плюс introspection: хорошо видно проблему и решение для массивов.

- В plain SQL-параметре `?` тип массива может быть неоднозначен.
- В `pg-schema` часто генерируется явный cast `?::typ[]`.
- Это делает поведение `ToField` для массивов стабильнее.
