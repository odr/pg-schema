# Ошибки компилятора как guide

Цель главы: научиться читать типовые compile-time ошибки `pg-schema` и быстро находить причину.

## Почему это важно

В `pg-schema` значимая часть ошибок ловится до запуска.  
Сначала это может выглядеть “шумно”, но сообщения реально помогают локализовать проблему.

## 1) Неизвестное поле или relation

Типичный сигнал:
- поле/связь не определены для таблицы в текущей схеме.

Обычно причина:
- опечатка;
- использовано имя таблицы вместо FK relation;
- ошибка в `Renamer`-ожиданиях.

Как действовать:
1. Проверить generated `Sch.hs`.
2. Сверить DB-имена в `qWhere/qPath`.
3. Упростить запрос до минимального рабочего.

## 2) Ошибка формы результата

Типичный сигнал:
- output-тип не согласован с root/path структурой.

Обычно причина:
- relation-поле ожидается как список, а в типе не список;
- nullable relation описан не как `Maybe`.

Как действовать:
1. Упростить выходной тип до 1-2 полей.
2. Постепенно добавить nested-поля обратно.

## 3) Ошибки tree DML ограничений

Типичный сигнал:
- для `insertJSON` не хватает mandatory-полей;
- для `upsertJSON` нет ни mandatory-набора, ни полного PK.

Как действовать:
1. Для каждого узла выписать mandatory и PK поля.
2. Проверить payload на каждом уровне дерева.

## 4) Ошибки глубины обхода (`annDepth`)

Типичный сигнал:
- достигнут предел relation walk depth.

Как действовать:
- увеличить `annDepth`;
- уменьшить глубину дерева в конкретной операции.

## Практический алгоритм чтения ошибок

1. Сначала найти имя таблицы/relation из сообщения.
2. Определить, это проблема schema name, поля или формы результата.
3. Упростить тип и запрос до минимума.
4. Вернуть сложность пошагово.

## Мини-checklist после главы

- [ ] Умеете отличить ошибку field/relation от ошибки shape.
- [ ] Умеете отлаживать через “сначала минимум, потом усложнение”.
- [ ] Понимаете, когда проблема в `annDepth`.
