# Edge cases и FAQ

## Nullable relation и `Maybe`

Если FK nullable, relation-поле в Haskell должно быть `Maybe`.
Если FK NOT NULL, relation не должен быть `Maybe`.

Это одна из самых частых причин type errors при описании выходного типа.

## Self-reference и `annDepth`

При рекурсивных связях глубина обхода ограничивается `annDepth` в `Ann`.
Если depth слишком мал, получите compile-time ошибку про relation walk depth limit.

Практика:
- сначала увеличить `annDepth`,
- если нужно — упростить глубину результата.

## Composite PK/FK

Для составных ключей важны:
- полный набор колонок ключа,
- корректный порядок и типы.

Частичный ключ для upsert не подходит.

## Returning как subtree input

Для tree DML возвращаемая структура должна быть поддеревом входной.
Если в `returning` есть ветка, которой нет во входе, это должно быть ошибкой.

## Что важно знать про `Unsafe*`

`UnsafeCol` и `pUnsafeCond` полезны для нетипичных SQL-случаев, но:
- обходят часть safety-проверок DSL;
- требуют аккуратного ревью;
- лучше использовать локально и минимально.

Правило: сначала попытайтесь выразить запрос через обычный типобезопасный DSL.

## Нужно ли регенерировать `Sch.hs` после изменения БД?

Да. Рабочий цикл: изменили схему -> запустили генератор -> собрали проект.

## Почему `qPath` “не находит” связь?

Обычно из-за имени relation:
нужно использовать имя FK constraint (с учетом `Renamer`), а не имя таблицы.

## Что делать с длинными type errors?

Практика:
1. Упростить output-тип до минимума.
2. Убрать часть `qPath`/условий.
3. Проверить `Sch.hs` (имена и типы).
4. Вернуть сложность по шагам.
